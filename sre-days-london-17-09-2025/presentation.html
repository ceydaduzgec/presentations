What is Terragrunt?

Terragrunt is a thin wrapper around Terraform that provides extra tools for:

  • Keeping your Terraform configurations DRY (Don't Repeat Yourself)
  • Managing multiple environments and configurations
  • Handling remote state and locking automatically
  • Managing dependencies between Terraform modules
  • Providing additional safety features and workflows

The Problems with Pure Terraform

  • Code Duplication: Same configurations copied across environments
  • Remote State Management: Manual backend configuration for each module
  • Environment Management: No native way to promote configs between envs
  • Dependency Management: Complex inter-module dependencies
  • Configuration Drift: Hard to maintain consistency across projects

Terraform vs Terragrunt

Aspect Terraform Terragrunt
Configuration Verbose, repetitive DRY, reusable
Remote State Manual setup Automatic generation
Multi-Environment Complex Built-in support
Dependencies Limited Advanced dependency management

Core Terragrunt Concepts

  • terragrunt.hcl: Configuration file that defines Terragrunt behavior
  • Remote State: Automatically configured backend storage
  • Inputs: Variables passed to Terraform modules
  • Dependencies: Define module execution order
  • Include: Inherit configuration from parent files
  • Generate: Create files dynamically (provider configs, etc.)

Typical Terragrunt Project Structure

infrastructure/ ├── terragrunt.hcl # Root configuration ├── dev/ │ ├── terragrunt.hcl # Dev environment config │ ├── vpc/ │ │ └── terragrunt.hcl # VPC module config │ └── app/ │ └── terragrunt.hcl # App module config ├── staging/ │ └── ... (similar structure) └── prod/ └── ... (similar structure)

Simple Terragrunt Configuration

# terragrunt.hcl terraform { source = "git::https://github.com/your-org/terraform-modules//vpc" } remote_state { backend = "s3" generate = { path = "backend.tf" if_exists = "overwrite_terragrunt" } config = { bucket = "your-terraform-state-bucket" key = "${path_relative_to_include()}/terraform.tfstate" region = "us-west-2" } } inputs = { vpc_cidr = "10.0.0.0/16" availability_zones = ["us-west-2a", "us-west-2b"] environment = "dev" }

Managing Dependencies

# app/terragrunt.hcl dependency "vpc" { config_path = "../vpc" } dependency "database" { config_path = "../rds" } inputs = { vpc_id = dependency.vpc.outputs.vpc_id subnet_ids = dependency.vpc.outputs.private_subnet_ids db_endpoint = dependency.database.outputs.endpoint environment = "dev" }

Terragrunt automatically handles the execution order!

DRY Configuration with Include

# Root terragrunt.hcl remote_state { backend = "s3" config = { bucket = "terraform-state-${get_aws_account_id()}" key = "${path_relative_to_include()}/terraform.tfstate" region = "us-west-2" } } # Child terragrunt.hcl include "root" { path = find_in_parent_folders() } terraform { source = "git::https://github.com/your-org/modules//vpc" }

Essential Terragrunt Commands

  • terragrunt plan → Same as terraform plan, but with Terragrunt features
  • terragrunt apply → Apply changes with dependency resolution
  • terragrunt run-all plan → Run plan across all modules in directory
  • terragrunt run-all apply → Apply all modules in correct order
  • terragrunt destroy → Destroy infrastructure
  • terragrunt validate → Validate configuration
  • terragrunt graph-dependencies → Visualize dependencies

Terragrunt Best Practices

  • Use remote modules: Keep Terraform modules in separate repositories
  • Environment separation: Separate directories for each environment
  • State isolation: Each module gets its own state file
  • Minimal duplication: Use include blocks for common configuration
  • Dependency management: Be explicit about inter-module dependencies
  • Version pinning: Pin module versions for production
  • Testing: Use terragrunt plan in CI/CD pipelines

Getting Started Today

Quick Start Steps:

  1. Install Terragrunt: brew install terragrunt
  2. Create your first terragrunt.hcl
  3. Define remote state configuration
  4. Add your first module with inputs
  5. Run terragrunt plan and terragrunt apply

Resources:

  • Official docs: terragrunt.gruntwork.io
  • Example configurations on GitHub

Key Takeaways 🎯

  • Terragrunt eliminates code duplication across environments
  • Automatic remote state management saves time and reduces errors
  • Built-in dependency management ensures proper deployment order
  • DRY principle makes infrastructure code maintainable
  • Easy multi-environment workflows from dev to production
  • Works seamlessly with existing Terraform modules

💡 Bottom Line: If you're managing multiple environments with Terraform, Terragrunt will save you time, reduce errors, and make your infrastructure code more maintainable.
